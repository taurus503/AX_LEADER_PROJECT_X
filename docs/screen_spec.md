# 삼성생명 민원 지식봇 화면기획서

## 1. 화면 목표

상담사 또는 민원/CS 기획 담당자가 고객 질문을 입력하면, 관련 FAQ를 근거로 참고용 답변 초안을 확인하고 고객 발송 전 담당자 확인 여부를 판단하는 단일 화면이다. 이 화면은 투자판단이나 확정 안내가 아니라, 민원 응대 초안 작성과 근거 확인을 돕는 내부 업무 보조 화면으로 설계한다.

## 2. 정보구조(IA)

상단에는 서비스명, 사용 범위, 면책 배너를 배치한다. 사용자는 화면을 열자마자 "실고객 정보 입력 금지", "답변은 참고용", "고객 발송 전 담당자 확인" 원칙을 먼저 인지해야 한다.

가장 중요한 영역은 질문 입력과 실행 버튼이다. 모바일에서는 상단 안내 아래에 바로 질문 입력창을 두고, 실행 버튼은 입력창 하단에 고정 폭으로 배치한다. 데스크톱에서는 입력 영역과 결과 영역을 2단으로 나누되, 읽기 순서는 질문 입력 → 처리 상태 → 답변 → 근거 → 담당자 확인 순서를 유지한다.

결과 화면에서는 답변 카드를 먼저 보여주고, 그 아래에 근거 FAQ 리스트를 접을 수 있는 영역으로 둔다. 근거는 업무 검토 시 필요하지만 첫 시선에서는 답변과 위험 여부가 더 중요하므로, 모바일에서는 FAQ 상세 내용을 접고 제목·분류·출처만 먼저 보여준다.

하단에는 담당자 확인 체크 영역을 둔다. "고객에게 바로 발송" 버튼은 두지 않고, "담당자 확인 필요" 상태와 확인 메모만 남기는 구조로 한다.

## 3. 핵심 흐름

1. 사용자가 질문 입력창에 고객 질문을 입력한다.
2. 입력 중 개인정보처럼 보이는 긴 숫자 패턴이 있으면 입력을 막거나 경고한다.
3. 실행 버튼을 누르면 로딩 상태에서 "FAQ 검색 중", "답변 초안 생성 중", "근거 확인 중" 단계를 순서대로 보여준다.
4. 결과가 있으면 답변 카드, 근거 FAQ 리스트, 담당자 확인 필요 배지를 보여준다.
5. 검색 결과가 없거나 금액·확정·범위 밖 질문이면 단정 답변 대신 오류·범위 밖 상태로 전환하고 담당자 확인을 안내한다.

## 4. 화면 상태 5종

### 초기

질문 입력창은 비어 있고, 빈 상태 문구로 "고객 질문을 입력하면 관련 FAQ를 찾아 답변 초안을 만듭니다."를 보여준다. 면책 배너는 항상 노출한다.

### 입력 중

실행 버튼이 활성화된다. 입력값 아래에는 "계약번호, 주민번호, 카드번호 등 개인정보는 입력하지 마세요." 안내를 유지한다. 개인정보 의심 숫자가 감지되면 버튼을 비활성화하고 "개인정보로 보이는 숫자는 삭제 후 다시 입력해 주세요."를 표시한다.

### 로딩

버튼은 비활성화하고 진행 문구를 표시한다. 예: "관련 FAQ를 찾고 있습니다." 로딩 중에도 질문 내용은 유지해 사용자가 맥락을 잃지 않게 한다.

### 결과

답변 카드에는 참고용 답변, 담당자 확인 필요 배지, 생성 기준 문구를 포함한다. 근거 FAQ 리스트에는 분류, 질문, 출처, 유사도 점수를 표시한다. FAQ 내용 상세는 모바일에서 기본 접힘, 데스크톱에서는 첫 1개만 펼침으로 둔다.

### 오류·범위 밖

검색 0건, 근거 부족, 금액/확정 판단, 개인정보 입력 의심, API 오류는 모두 단정하지 않는다. 문구는 "근거가 충분하지 않아 답변 초안을 만들 수 없습니다. 담당자 확인이 필요합니다."로 통일해 신뢰감을 유지한다.

## 5. 컴포넌트

| 컴포넌트 | 역할 | 주요 상태 |
|---|---|---|
| 질문 입력 | 고객 질문 입력, 개인정보 의심 숫자 차단 | 기본, 입력 중, 차단 |
| 실행 버튼 | FAQ 검색과 답변 생성 시작 | 비활성, 활성, 로딩 |
| 면책 배너 | 공개/더미 데이터, 참고용 답변, 사람 확인 원칙 안내 | 항상 표시 |
| 답변 카드 | 답변 초안과 담당자 확인 배지 표시 | 결과, 범위 밖 |
| 근거 FAQ 리스트 | FAQ 분류·질문·출처·점수 표시 | 접힘, 펼침 |
| 담당자 확인 필요 배지 | 고객 발송 전 확인 필요 여부 표시 | 일반, 경고 |
| 담당자 확인 영역 | 확인 체크와 내부 메모 입력 | 미확인, 확인됨 |

## 6. UX 라이팅

### 버튼

- 기본: `답변 초안 만들기`
- 로딩: `확인 중...`
- 차단: `개인정보 삭제 필요`

### 플레이스홀더

`예: 보험금 청구에 필요한 서류가 무엇인지 문의드립니다.`

### 빈 상태

`질문을 입력하면 관련 FAQ를 찾고, 근거가 있는 참고용 답변 초안을 만듭니다.`

### 개인정보 차단

`개인정보로 보이는 긴 숫자가 포함되어 있습니다. 계약번호, 주민번호, 카드번호는 삭제해 주세요.`

### 오류·범위 밖

`근거가 충분하지 않아 답변 초안을 만들 수 없습니다. 담당자 확인이 필요합니다.`

### 면책 배너

`이 화면은 공개·더미 FAQ 기반의 참고용 답변 초안 도구입니다. 고객 발송 전 담당자 확인이 필요합니다.`

## 7. 접근성

색 대비는 본문 텍스트 4.5:1 이상, 배지와 경고 영역은 색상만으로 의미를 전달하지 않고 아이콘 또는 텍스트를 함께 사용한다. 키보드 순서는 면책 배너 → 질문 입력 → 실행 버튼 → 답변 카드 → 근거 FAQ → 담당자 확인 영역 순서로 둔다.

질문 입력창에는 명확한 `label`을 제공하고, 오류 문구는 `aria-live` 영역에 노출한다. 로딩 상태는 `aria-busy`와 텍스트 진행 상태로 제공한다. 모바일에서는 한 손 조작을 고려해 실행 버튼 높이를 44px 이상으로 한다.

## 8. 모바일 우선 레이아웃

모바일은 단일 열 구조를 기본으로 한다. 상단 안내, 질문 입력, 결과 카드, 근거 FAQ, 담당자 확인 순서로 세로 배치한다. 답변과 담당자 확인 배지는 한 화면 안에서 먼저 보이게 하고, 근거 상세는 접어서 스크롤 부담을 줄인다.

데스크톱에서는 입력/상태 영역을 왼쪽, 결과/근거 영역을 오른쪽에 배치할 수 있다. 다만 화면 흐름과 문구는 모바일과 동일하게 유지한다.

## 9. 개인정보 유사 숫자 차단 기준

정적 화면 뼈대에서는 다음 패턴을 우선 차단한다.

- 숫자 6자리 이상 연속 입력
- 하이픈 포함 숫자 묶음이 10자리 이상으로 보이는 경우
- 주민번호처럼 보이는 `######-#######` 형식
- 카드번호처럼 보이는 `####-####-####-####` 형식

차단 시 검색·답변 로직은 실행하지 않고, 입력 수정 안내만 보여준다.

## 10. 구현 범위

이번 `index.html` 뼈대에는 검색·답변 API 로직을 비워둔다. 화면 상태 전환, 개인정보 의심 숫자 차단, 샘플 결과 표시용 마크업만 포함한다. 실제 Supabase 검색과 BizRouter 답변 생성은 이후 `/api/answer` 또는 별도 클라이언트 연동 단계에서 붙인다.

## 11. 시각 디자인 방향

TinyFaces 레퍼런스의 섹션 리듬을 민원 지식봇 업무 화면에 맞게 재해석한다. 상단에는 얇은 내비게이션과 강한 브랜드 마크를 두고, 첫 화면은 히어로 일러스트와 실제 질문 입력 패널이 동시에 보이도록 구성한다. 이후에는 러닝 텍스트 띠, 3개 기능 카드, 짙은 파란색 상태 보드, FAQ 아코디언, 하단 CTA로 이어지게 한다.

시각 언어는 네오브루탈리즘을 사용한다. 두꺼운 검정 외곽선, blur 없는 hard offset shadow, 종이색 배경, 파랑·노랑·코랄·민트의 평면 색을 사용한다. 단, NFT/캐릭터 수집 맥락은 사용하지 않고 보험·금융 민원 상담의 신뢰감과 업무 보조 성격을 우선한다.

구체 규칙은 `docs/design.md`를 따른다. 폰트는 Pretendard 계열을 우선 사용하고 제목은 800 weight로 둔다. `--ink #0a0a0a`, `--paper #f8f5ec`, `--blue #0d63c9`를 기본 토큰으로 사용하며 노랑, 코랄, 민트를 포인트 색으로 제한한다. 카드, 버튼, 입력칸, 히어로 이미지는 3px 검정 테두리와 8px 8px 0 hard shadow를 적용한다. `담당자 확인 필요`는 코랄 배지로 표시한다. 과한 애니메이션은 금지하고, 모바일은 1열 구조를 유지한다.
